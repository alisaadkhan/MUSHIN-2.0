## ADR-029: Identity Resolution Matching Algorithm

**Status:** Proposed
**Context:** DOC-018 B1 and DOC-014 named this as an unspecified hard problem. The schema (`gcp.creator.merge_status`, `merge_confidence`) stores the *output* of identity resolution but no document defined the *method*. This ADR defines it, per the spec provided in the Master Implementation Directive.

### Decision

A **deterministic, weighted-evidence scoring model** — not a black-box classifier — because the constraint is explainability and auditability, and every merge (especially auto-merges) must be defensible to a human reviewer after the fact.

**Signal weights** (summed, capped at 100, each contributing signal recorded in `evidence_breakdown`):

| Signal | Weight | Rationale |
|---|---|---|
| Shared verified email or WhatsApp number | 45 | Near-unfakeable; two accounts controlling the same verified contact channel is the strongest available evidence |
| Verified website ownership match (shared domain + matching ownership metadata) | 35 | Hard to fake at both ends |
| Explicit cross-mention claiming ownership (bio link, "this is my other account") | 35 | Self-asserted linkage from the account itself |
| Shared Linktree / link-in-bio exact URL | 20 | Common but not unique — some link-in-bio pages are agency-managed, not creator-owned |
| Face similarity score above 0.92 | 20 | Supporting signal only — deliberately capped low given photo-condition variance and fairness concerns with facial-similarity matching; never allowed to be decisive alone |
| Username identical/near-identical across platforms | 15 | Common convention, weak alone (many creators intentionally pick matching handles; so do impersonators) |
| Cross-mentioned handles (mentioned in a caption/comment, not a bio claim) | 10 | Weak positive signal |
| Structured metadata match (e.g., identical listed business contact) | 10 | Weak positive signal |
| Display name exact match | 10 | Weakest — highly collision-prone (common names) |
| Manual reviewer confirmation | Overrides to 100 | A human already did the work; no need to re-derive |

**Thresholds** (as specified): `≥90` → auto-merge (`merge_status = active`, linked). `60–89` → `candidate`, `human_review_required = true`. `<60` → stays `independent`; weak signals are still logged for future re-evaluation as more data arrives, not discarded.

**Output schema** (as specified): `merge_confidence` (float 0–100), `confidence_reasoning` (string, human-readable summary), `evidence_breakdown` (array of `{signal, weight, detail}`), `human_review_required` (boolean).

**Design rule:** no single signal below 35 points may reach the auto-merge threshold on its own — auto-merge requires either one strong signal (verified contact or website ownership) or multiple corroborating weaker signals. This prevents a single spoofable or common-coincidence signal (matching username, matching display name) from ever triggering an unreviewed merge by itself.

### Minor-Safety Default (part of this decision, not a separate policy)

Any evidence source that surfaces an age signal — self-disclosed age under 18 in bio/caption text, a platform's own child-directed/parental-account designation, or explicit "kids' content" framing — sets `minor_signal = true` on the resulting creator record, independent of `merge_confidence`.

`minor_signal = true` gates the following closed **by default, regardless of workspace or subscription tier**, until a real child-creator policy is ratified:
- Contact reveal (`wp.reveal`)
- Campaign-add / outreach sequence enrollment
- Any automated commercial-outreach trigger

Discovery/search results and read-only intelligence (niche classification, audience estimates) are unaffected — this default only blocks the *commercial contact* surface, since that's the specific risk of processing profiles that may belong to minors. This is a stopgap default, not the actual policy (age signals from public content are unreliable and this is deliberately over-inclusive) — it exists so the pipeline isn't blind to the question while DOC-028's open item is resolved.

**Consequences:** identity resolution can ship without waiting on a full child-safety policy, because the commercially-risky surface is closed by default rather than open by default. Reversing this default requires a real product/legal decision (Decision Authority Matrix: "legal implications," "user-visible workflow changes" — human approval required), not a config flip.

## ADR-030: Credit Reservation ↔ Subscription-State Interaction

**Status:** 🟡 PROPOSED — requires human sign-off. This is a subscription-behavior change per your own Decision Authority Matrix; it is written up here as a recommendation, not treated as decided.

**Context:** DOC-018 E4 (gap). Schema already anticipates this (`reservation_status_enum` includes `expired`; `credit_reservation.resolution_reason` includes `subscription_expired`) but no document specifies the actual rule.

### Recommended Decision

1. **Subscription cancellation/expiry does not force-release in-flight reservations.** No new webhook-triggered release path. The existing TTL sweeper (PATCH-005, 30-minute default) remains the *only* release mechanism, for both the normal case and the post-cancellation case. Rationale: a second, webhook-triggered release path racing against normal job completion is exactly the kind of special-case complexity that causes the bugs RB-01/RB-02 exist to catch.
2. **Jobs already holding a reservation at cancellation time run to completion or natural TTL expiry**, and results are delivered normally. The backend cost (Apify/LLM calls) is already sunk; withholding results after the fact helps no one and creates a support-ticket-generating experience.
3. **What does change at cancellation:** no *new* reservations may be created once the workspace's Paddle subscription leaves an active state — enforced at the existing API eligibility check (DOC-020), not in the reservation/ledger layer itself.
4. Ledger accounting is unaffected: reservations still resolve via `completed` or `expired` exactly as they do today; `subscription_expired` remains available as a `resolution_reason` for the sweeper to set if it's the operative cause, for accurate reporting.

**Why this needs sign-off, not just implementation:** an alternative, equally defensible design (immediately force-release all reservations on cancellation, refuse to deliver results from jobs that outlive the cancellation) has real revenue and customer-experience trade-offs that are a product decision, not an engineering one. The design above is a recommendation optimized for simplicity and customer fairness, not the only valid answer.

## ADR-031: Queue-Class-to-Infrastructure Mapping

**Status:** Proposed
**Context:** DOC-016 defines abstract queue classes; DOC-022 implements concrete SQS queues. They don't map 1:1 (F-09 in the findings report) — DOC-016 has no `erasure` class even though `q-erasure` exists in infra, and DOC-016's `webhooks` class has no corresponding named queue.

### Decision

| Abstract class (DOC-016) | Concrete queue (DOC-022) | Note |
|---|---|---|
| `interactive` | `q-discovery-high` | Direct match — user-waiting, highest priority |
| `discovery-bulk` | `q-discovery-standard` | Direct match — live search fan-out |
| `scheduled` | `q-rescore-low` | Direct match — PATCH-010 re-scoring |
| `events` | `q-outbox-relay` | Direct match — outbox relay is the event-transport mechanism |
| `webhooks` | *(no dedicated queue — synchronous)* | Inbound webhook **verification** (signature check, raw-payload write to `paddle_webhook_raw`) happens synchronously at the gateway and acks immediately. Any async follow-on work re-enters the system as a normal domain event and routes through `events`/`q-outbox-relay` like anything else — it doesn't need its own queue. |
| *(new)* `erasure` | `q-erasure` | `q-erasure` already existed in infra without a named abstract class. Formalizing it here: erasure jobs get their own class specifically because they have a distinct, hard SLA (72-hour regulatory clock) that shouldn't compete with or be starved by discovery/rescoring volume. |

**Amendment to DOC-016:** add `erasure` as a sixth queue class alongside the original five.
**Amendment to DOC-022:** no infra change needed — `q-erasure` already exists; this ADR just gives it a documented home in the abstract taxonomy.
