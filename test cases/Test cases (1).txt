This is the Production Verification Matrix. It is not a test plan; it is a proof of survivability. We do not test to confirm the system works when everything is fine. We test to confirm the system degrades gracefully, heals automatically, and fails safely when the underlying infrastructure turns hostile.

# PRODUCTION VERIFICATION MATRIX

## 1. DNS / Global Traffic

**[Functional] Active-Active Routing**
*   **Obj:** Verify traffic routes to the closest healthy region.
*   **Pre:** Multi-region deployment active.
*   **Steps:** Query DNS from simulated geo-IPs; compare resolved IPs to regional fleet.
*   **Exp:** Traffic resolves to the local regional endpoint.
*   **Metrics:** DNS query latency, geo-routing accuracy.
*   **Pass:** 100% routing accuracy to designated local region.
*   **Sev:** P2 (Misrouted traffic increases latency).
*   **Auto:** Fully automated in CI.

**[Failure Injection] Primary Region Health Check Failure**
*   **Obj:** Verify DNS failover triggers within TTL.
*   **Pre:** Region A active, Region B standby.
*   **Steps:** Block health check probes at the edge of Region A via local firewall.
*   **Exp:** DNS stops returning Region A IPs within 60s; returns Region B.
*   **Metrics:** Time to failover, DNS TTL compliance.
*   **Pass:** Failover completes < 90s. Zero manual intervention.
*   **Sev:** P1 (Complete regional outage if failed).
*   **Auto:** Automated via Chaos Monkey in isolated staging.

**[Chaos] Stale DNS Cache Poisoning Simulation**
*   **Obj:** Ensure clients do not remain pinned to dead IPs.
*   **Pre:** App implements connection timeouts and DNS re-resolution logic.
*   **Steps:** Inject a route blackhole for a specific app node IP at the router level.
*   **Exp:** Client libraries detect TCP failure, trigger DNS re-resolution, and route to healthy node.
*   **Metrics:** Time to re-establish connection.
*   **Pass:** Zero stuck connections; reconnect < 5s.
*   **Sev:** P1.
*   **Auto:** Automated network chaos (e.g., Toxiproxy).

**[Load] DNS Query Flood**
*   **Obj:** Ensure DNS provider does not rate-limit or drop legitimate resolves.
*   **Pre:** Alerting configured for DNS throttling.
*   **Steps:** Generate 10x normal DNS query volume from distributed IPs.
*   **Exp:** DNS provider handles load; app connection rate remains stable.
*   **Metrics:** DNS throttle rate (429s from provider).
*   **Pass:** 0% throttle rate from provider.
*   **Sev:** P2.
*   **Auto:** Automated load test.

**[Security] DNSSEC Bypass Attempt**
*   **Obj:** Verify DNSSEC validation is enforced.
*   **Steps:** Attempt to serve forged DNS responses without valid RRSIG records.
*   **Exp:** Resolvers drop forged responses; SERVFAIL returned.
*   **Metrics:** DNSSEC validation failure rate.
*   **Pass:** 100% rejection of unsigned/modified zones.
*   **Sev:** P1 (Traffic hijacking risk).
*   **Auto:** Automated security scanning.

**[Recovery] DNS Zone Accidental Deletion**
*   **Obj:** Validate zone recovery process.
*   **Steps:** Delete a non-critical test sub-zone from the provider.
*   **Exp:** IaC pipeline detects drift (or manual restore from version control) and reapplies within 5 minutes.
*   **Metrics:** Time to detection, Time to restoration.
*   **Pass:** Restoration < 5 mins.
*   **Sev:** P1.
*   **Auto:** Semi-automated (requires IaC apply).

**[Compliance] Audit Log Continuity During Failover**
*   **Obj:** Ensure no DNS routing changes are made without audit.
*   **Steps:** Trigger a failover; inspect audit logs.
*   **Exp:** Every TTL change, health check toggle, and zone update is logged with actor ID.
*   **Metrics:** Audit log coverage.
*   **Pass:** 100% of infrastructure mutations logged.
*   **Sev:** P2.
*   **Auto:** Automated compliance assertion.

**[SLO Validation]**
*   **SLI:** Percentage of successful DNS resolutions (non-SERVFAIL, non-NXDOMAIN for valid records).
*   **SLO:** 99.99% availability.
*   **Error Budget:** 4.32 minutes/month.
*   **Alert Thresholds:** > 0.01% failure rate over 5m window.

**[Runbook] Global Traffic Failover**
*   **Detection:** DNS provider alerting + surge in 5xx from edge nodes.
*   **Diagnosis:** Check regional health dashboards; confirm widespread regional failure (not isolated node).
*   **Mitigation:** Manually promote secondary region to primary in DNS provider (if automatic failover stalled).
*   **Rollback:** Revert DNS weights once primary region is healthy and verified.
*   **Communication:** Status page update ("Investigating DNS routing issues").
*   **Postmortem:** Why did automatic failover fail? Why did the region go down?
