- **Temporal workflows**: For durable background jobs like document ingestion (Layer 7)

### Long-term (10,000+ users)
- **Multi-region deployment**: Active-passive for writes, active-active for reads
- **Sharded databases**: Tenant-based sharding for Postgres (Layer 9)
- **Dedicated inference clusters**: GPU autoscaling with fair-share scheduling (Layer 12)
- **Cell-based architecture**: Self-contained deployments for tenant subsets (Stage 4)

## Security Considerations for Your Context

Given your interest in security architecture:

- **Row-Level Security**: Postgres RLS as defense-in-depth for tenant isolation (Layer 9)
- **Audit logs**: Append-only WORM storage for all auth events and data access (Layer 14)

## Reliability Patterns to Implement Now

The document emphasizes these patterns uniformly across layers:

2. **Circuit breakers**: Per-dependency, not global
3. **Dead Letter Queues**: Monitor DLQ growth as P1 alerts
4. **Graceful degradation**: Document fallback paths for every feature

## Cost Optimization Insights

For AI SaaS specifically:

- **Semantic caching**: Cache identical prompt completions (with disclosure)
- **Model routing**: Route easy queries to smaller models
- **Quantization**: Use PQ/SQ/binary quantization for vector indexes
- **Batch inference**: Batch embedding generation at scale
- **Multi-provider routing**: Cost-aware fallback between LLM providers
